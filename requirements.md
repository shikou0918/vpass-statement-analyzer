# Vpass明細分析アプリ 要件定義書

## 1. 背景・目的

### 現状の課題

- Vpassからクレジットカード利用明細をCSVでエクスポートできるが、そのままでは支出傾向を把握しづらい。
- 既存の家計簿アプリは高機能だが、口座連携、アカウント登録、クラウド保存などが前提になりやすい。
- 自分のカード明細だけを、ローカル環境で安全に、素早く分析したい。
- Vpass CSVはヘッダー行がなく、文字コードもCP932/Shift_JIS系のため、一般的なCSVビューアでは扱いづらい場合がある。

### 本システムの導入目的

Vpassから取得したCSV明細を取り込み、月別・利用先別・カテゴリ別に支出を可視化する、自分専用の明細分析Webアプリを作成する。

### 期待される効果

- 毎月のカード支出総額をすぐ確認できる。
- セブン-イレブン、Suica、スーパー、自販機など、利用先ごとの支出傾向を把握できる。
- 利用先名に基づくカテゴリ分類ルールを蓄積できる。
- 明細データを外部サービスへ送信せず、ローカルで管理できる。
- 家計簿アプリより軽く、自分の分析目的に合わせて拡張できる。

## 2. ターゲットユーザー

### プライマリユーザー

- Vpassを利用している個人ユーザー。
- クレジットカード明細をCSVでダウンロードし、自分の支出傾向を把握したい人。
- 家計簿アプリの口座連携やクラウド保存に抵抗があり、明細データをローカルで管理したい人。
- Excelやスプレッドシートで毎月集計しているが、カテゴリ分類やグラフ化を自動化したい人。

### セカンダリユーザー

- 三井住友カード/Vpass利用者で、カード明細だけをシンプルに分析したい人。
- フリーランスや副業ユーザーで、クレジットカード明細から経費候補を確認したい人。
- 個人開発者・エンジニアで、自分の金融データを自由に加工・分析したい人。

### 初期開発における想定ユーザー

- 主な利用者は本人。
- PCブラウザでの利用を想定する。
- 外部公開、マルチユーザー、スマートフォンアプリ化は初期スコープ外とする。
- 技術検証と実用を兼ね、自分の実データを継続的に取り込んで改善していく。

## 3. 対象データ

### 入力ファイル

- Vpassからエクスポートしたクレジットカード利用明細CSV。
- 例: `202605.csv`
- 文字コード: CP932/Shift_JIS系を想定。
- ヘッダー行なし。
- 1行13列の固定形式を初期想定とする。

### 想定カラム

| 列番号 | 項目名 | 例 | 初期利用 |
| :--- | :--- | :--- | :--- |
| 1 | 利用日 | `2026/4/30` | 使用する |
| 2 | 利用先 | `セブン－イレブン` | 使用する |
| 3 | 利用者 | `ご本人` | 使用する |
| 4 | 支払方法 | `1回払い` | 使用する |
| 5 | 予備項目 | 空欄など | 保持のみ |
| 6 | 支払月 | `'26/05` | 使用する |
| 7 | 利用金額 | `213` | 使用する |
| 8 | 請求金額 | `213` | 使用する |
| 9-13 | 予備項目 | 空欄など | 保持のみ |

## 4. 機能要件

### 4.1 CSVインポート

- ユーザーはVpass CSVファイルをアップロードできる。
- アプリはCP932/Shift_JISのCSVを正しく読み込める。
- ヘッダーなしCSVとして、列番号ベースで項目をマッピングする。
- インポート前に読み込み結果のプレビューを表示する。
- 不正な行、列数不足、金額変換エラーを検出して表示する。
- 同じCSVを再インポートしても重複登録しない。
- CSV保存時は保存済みの自動分類ルールを未分類明細へ自動適用する。

### 4.2 明細一覧

- 利用日、利用先、利用者、支払方法、支払月、利用金額、請求金額、カテゴリを一覧表示する。
- 年月、利用先、カテゴリ、金額範囲で絞り込みできる。
- 利用先名でキーワード検索できる。
- 日付、金額、利用先で並び替えできる。

### 4.3 ダッシュボード

- 対象月の支出合計を表示する。
- 前月比を表示する。
- 利用先別支出ランキングを表示する。
- カテゴリ別支出内訳を表示する。
- 日別支出推移を表示する。

### 4.4 カテゴリ管理

- カテゴリを作成、編集、削除できる。
- カテゴリには名前と色を設定できる。
- 明細ごとにカテゴリを手動設定できる。
- 利用先名に基づく分類ルールを作成、編集、削除できる。
- 例: 利用先に `セブン` を含む場合は `コンビニ`。
- 同一条件、同一利用先パターン、同一カテゴリの分類ルールは重複作成できない。
- 未分類の利用先を件数順に集約し、カテゴリ選択から分類ルールを作成できる。ルール作成・編集後は一致する既存明細へ自動適用し、既存カテゴリも新しいルールに合わせて更新する。
- 既存明細に分類ルールを再適用できる。

### 4.5 分析機能

- 月別支出推移を表示する。
- 利用先別の月次推移を表示する。
- カテゴリ別の月次推移を表示する。
- 固定費候補を抽出する。
- 少額高頻度の支出候補を抽出する。

### 4.6 データ管理

- 取り込んだ明細をローカルDBに保存する。
- CSVファイル単位でインポート履歴を管理する。
- 明細データをCSVでエクスポートできる。
- 必要に応じてインポート済みファイル単位で削除できる。
- インポート済みファイルを削除すると、そのファイル由来の明細、マッピング、インポートエラーを削除する。
- インポート済みファイルを削除した後は、同一内容のCSVを再インポートできる。
- インポート済みファイル削除は、対象ファイル名と影響範囲を確認してから実行する。

## 5. 非機能要件

### セキュリティ・プライバシー

- 明細データは外部サーバーへ送信しない。
- 初期版ではローカル環境でのみ動作させる。
- 認証機能は初期スコープ外とする。
- 将来的にクラウド化する場合は、認証、暗号化、バックアップ方針を再設計する。

### パフォーマンス

- 数千件程度の明細を快適に表示・検索できる。
- CSVインポートは数秒以内に完了することを目標とする。
- 明細一覧はページングまたは仮想スクロールを検討する。

### 保守性

- Vpass CSVの列定義をコード内に閉じ込めず、インポーターとして分離する。
- 将来的にPayPay、銀行CSV、他カードCSVを追加できる設計にする。
- 分類ルールは明細データとは独立して管理する。

### 動作環境

- macOS上のローカル開発環境。
- PC版ChromeまたはEdgeの最新版。
- 初期版はスマートフォン対応を必須としない。

## 7. 技術構成案

| 領域 | 候補 | 理由 |
| :--- | :--- | :--- |
| フロントエンド | Next.js / React | ローカルWebアプリとして作りやすく、画面追加もしやすい |
| スタイリング | Tailwind CSS | ダッシュボードUIを素早く作れる |
| CSVパース | Papa Parse + Encoding処理 | CSV処理と文字コード対応が必要 |
| データ保存 | SQLite | 自分用ローカルアプリに向いている |
| ORM | Prisma | SQLiteとの相性が良く、型安全に扱える |
| グラフ | Recharts | Reactで月次・カテゴリ別グラフを作りやすい |

## 9. MVPスコープ

初期開発では以下に絞る。

1. Vpass CSVをアップロードする。
2. CP932/Shift_JISとして読み込む。
3. 明細をSQLiteに保存する。
4. 明細一覧を表示する。
5. 月別支出合計を表示する。
6. 利用先別ランキングを表示する。
7. カテゴリを手動設定できる。
8. 利用先名に基づくカテゴリ自動分類ルールを作れる。

## 10. 初期スコープ外

- 銀行口座や証券口座との自動連携。
- Vpassへの自動ログインやスクレイピング。
- スマートフォンアプリ化。
- 複数ユーザー対応。
- クラウド同期。
- 外部AI APIによる自動分類。
- レシート読み取り。
- 予算管理。

## 11. 今後の確認事項

- 金額は利用金額と請求金額のどちらを主集計に使うか。
- 支払月ベースで見るか、利用日ベースで見るか。
- Suicaチャージなどを交通費にするか、電子マネーチャージとして別管理するか。
- コンビニ、スーパー、交通、外食、サブスクなど、初期カテゴリをどう分けるか。
- ローカルWebアプリとして始めるか、将来的なデスクトップアプリ化も考えるか。

## 12. 推奨開発ステップ

1. サンプルCSVを読み込むパーサーを作る。
2. 取り込みプレビュー画面を作る。
3. SQLiteに保存する。
4. 明細一覧画面を作る。
5. 月別合計と利用先ランキングを作る。
6. カテゴリ管理を追加する。
7. 分類ルールを追加する。
8. グラフと分析画面を追加する。
